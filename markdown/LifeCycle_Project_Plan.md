# [기획안] 일상 사이클 관리 서비스 (가칭: LifeCycle)

## 1. 서비스 개요
* **컨셉:** 자동차 소모품 관리 시스템을 일상생활 전반으로 확장한 '역방향 일정 관리' 서비스.
* **핵심 가치:** "미래의 할 일"이 아닌 **"과거의 마지막 수행일"**로부터 경과된 시간을 추적하여 적절한 루틴 유지를 도움.
* **대상:** 미용, 가전 필터 교체, 쇼핑 주기, 자동차 정비 등 반복되는 일상을 체계적으로 관리하고 싶은 사용자.

## 2. 주요 기능 (Functional Requirements)

### ① 항목별 주기 설정 및 대시보드
* **항목 생성:** 관리할 대상(예: 이발, 주유, 옷 구입, 칫솔 교체 등) 등록. 생성 시 **별도 테이블로 관리되는 카테고리**를 참조하여 선택.
* **항목 수정:** 항목명, 목표 주기, 카테고리, 알림 설정 등 기본 정보 수정.
* **항목 아카이브:** 삭제 대신 보관. `ManagementItem.status` 플래그로 비활성 처리(대시보드 기본 목록에서 제외, 복구 가능).
* **권장 주기 설정:** 사용자에게는 **년 / 개월 / 주 / 일** 단위로 입력·표시. 내부 계산은 **일(day) 단위**로 통일.
* **D-Day 및 경과일 표시:**
    * **기준일:** 해당 항목의 **가장 최신 수행 로그(`performedAt` 최대값)** 기준.
    * **경과 시간(Elapsed Time):** "마지막으로 수행한 지 며칠 지났는가" 표시.
    * **잔여 시간(Remaining Time):** "다음 예정일까지 며칠 남았는가" 표시.
* **날짜·시간 기준:** 모든 일(day) 계산은 **각 일 0시(자정) 기준**. 시·분·초는 비교에서 제외.

### ② 히스토리 로그 (Log Management)
* **원클릭 기록:** 오늘 수행했을 경우 버튼 한 번으로 날짜 업데이트.
* **과거 기록 수정:** 실제 수행일과 기록일이 다를 경우 날짜 소급 적용.
* **메모 기능:** 간단한 텍스트 기록 (예: 주유 시 리터당 가격, 옷 구매 시 브랜드 등).

### ③ 상태 시각화 (Status Visualization) — MVP 기본 UI
대시보드의 **기본 UI 구성 요소**로 1단계(MVP)부터 포함한다.
* **신호등 시스템:**
    * **안전(녹색):** 수행한 지 얼마 안 된 상태(목표 주기의 80% 미만 경과).
    * **주의(황색):** 설정한 주기의 80% 이상 경과.
    * **지연(적색):** 설정한 주기를 초과함.
* **진행바(Progress Bar):** 현재 주기의 어느 지점에 있는지 시각적으로 표현.
* **특수 표시 (화면 기획에서 상세 정의):**
    * **주기 0일(당일 수행):** "오늘 수행함" 등 텍스트·아이콘·이미지로 대체 표시.
    * **수행 로그 없음:** empty-state 이미지·아이콘·픽토그램 또는 안내 문구로 처리.

### ④ 통계 및 인사이트
* **평균 주기 계산:** 사용자의 실제 기록 데이터를 바탕으로 평균 수행 주기 산출.
* **비용 추적:** 항목별 지출 금액 입력 시 월별/항목별 총지출 통계 제공.

## 3. 카테고리 구성 예시
카테고리는 **별도 테이블(`Category`)**에서 관리하며, 항목 생성·수정 시 목록에서 선택한다. 초기 시드 예시는 아래와 같다.

* **그루밍:** 미용실, 네일, 스케일링, 화장품 개봉.
* **모빌리티:** 주유, 엔진오일, 와이퍼, 타이어 로테이션.
* **리빙:** 정수기 필터, 에어컨 청소, 침구 세탁, 칫솔 교체.
* **소비:** 의류 구입, 운동화 구입, 영양제 구매.

## 4. 시스템 구조 (Technical Architecture)

### [데이터베이스 설계]
| 테이블 | 필드 구성 | 비고 |
| :--- | :--- | :--- |
| **Category** | id, name, icon, userId, createdAt, updatedAt | 항목 생성 시 참조. User 테이블은 고도화 시 추가 |
| **ManagementItem** | id, userId, name, targetCycleDays, categoryId, status, notificationEnabled, createdAt, updatedAt | `targetCycleDays`: 일 단위 저장. `status`: active / archived |
| **ActivityLog** | id, userId, itemId, performedAt, cost, note, createdAt, updatedAt | 항목별 다건. 대시보드는 **최신 performedAt** 기준 |

**User / Auth:** 고도화 단계에서 User·Auth 테이블 및 Next.js 인증 연동 시 함께 설계·구현. 별도 기획 문서로 정리.

**고도화 전 개발·테스트용 식별자:**
* 인증 도입 전까지 모든 Category / ManagementItem / ActivityLog의 `userId`는 **마스터·테스트 계정 값 `1`** 로 고정하여 구조만 맞춘다.
* DB 전환 후에도 동일 필드로 사용자별 데이터 격리가 가능하도록 미리 반영한다.

**알림(Notification):**
* 실제 발송은 **클라우드 DB 운용 후**, 정해진 시각에 **크론(스케줄러)** 으로 `notificationEnabled === true` 인 항목을 조회·푸시하는 방식으로 고도화 단계에서 구현한다.
* MVP·localStorage 단계에서는 **발송 로직 없이 필드만 선반영**한다. 추후 구현 시 스키마 누락·오류를 방지하고, 문서만으로도 의도를 파악할 수 있게 한다.
* MVP에 포함할 필드(안): `notificationEnabled` (boolean). 고도화 시 추가 예정: `notificationTime` (HH:mm), 채널·푸시 토큰 등 — 상세는 고도화 기획 문서에서 확정.

**데이터 격리:**
* `ManagementItem`, `ActivityLog`, `Category` 모두 `userId`를 보유한다. 고도화 후 조회·변경 API는 **반드시 현재 세션 사용자의 `userId`로 필터**하여 타 사용자 데이터가 노출·공유되지 않도록 한다.
* **User 테이블은 고도화 전까지 생성하지 않는다.** 카테고리에는 확장을 위해 `userId`만 두고, 고도화 전까지 `1`(마스터·테스트)로 사용한다.

**리비전(Revision) 이력:** 도입하지 않음. `createdAt` / `updatedAt` 만으로 충분하며, 리비전 테이블은 용량·복잡도만 증가시킨다.

### [기술 스택]
* **Frontend:** Next.js 14+ (App Router), TypeScript, Tailwind CSS — 인증·멀티 디바이스 확장을 고려한 기본 프레임워크.
* **MVP 저장소:** 브라우저 localStorage + Mock API 서비스 계층 (상세는 개발 명세서 참고).
* **고도화 저장소:** Supabase 또는 PostgreSQL 기반 클라우드 DB.
* **배포:** 정적 빌드(`out/`), `DEPLOY_BASE_PATH`로 서브경로 설정 (README 참고).

### [비기능 요구 — 확장 고려]
* **접근성(a11y):** 키보드 탐색, 시맨틱 마크업, 색상 대비(신호등은 색만으로 정보 전달하지 않도록 아이콘·텍스트 병행) 등 WCAG 지향 설계를 MVP부터 의식한다.
* **다국어(i18n):** UI 문구·날짜·주기 단위(년/개월/주/일) 표기를 하드코딩하지 않고, 추후 locale 확장이 가능한 구조(예: next-intl 등)를 전제로 한다. 1차 언어는 한국어.
* **인증·멀티 디바이스:** 고도화 시 Auth 연동과 함께 **신규 기능**으로 추가. Next.js 기반으로 진행.

### [의도적으로 MVP 기획서 범위 외]
* **첫 사용자(Empty) UX:** 화면 기획 문서에서 플로우·카피·일러스트 상세 정의.
* **주기 0일·로그 없음 UI:** 화면 기획 문서에서 시각 요소(이미지·픽토그램 등) 상세 정의. 본 문서 §2-③에서 방향만 명시.

## 5. 단계별 개발 로드맵
1.  **1단계 (MVP):** 카테고리 참조 항목 등록·수정·아카이브, 로그 기록, 경과일·잔여일, **신호등·진행바(기본 UI)**, localStorage + `userId=1` 구조.
2.  **2단계 (통계):** 비용 입력 및 항목별·월별 비용 집계, 실제 평균 수행 주기 통계 — **`/stats` 화면** (localStorage 기준).
3.  **3단계 (고도화):** 클라우드 DB 이관, Auth·User(별도 기획 문서), 멀티 디바이스 동기화, 알림(크론·푸시), PWA 설치형 앱.
